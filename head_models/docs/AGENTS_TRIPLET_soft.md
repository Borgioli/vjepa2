# Future Codex Guide: Triplet Soft Refinement

This document is the working reference for the Tool5 **soft triplet refinement** pipeline in `/path/to/phase_triplet_heads_bundle/vjepa2_1`.

The goal is to make it very clear which files matter, how the model works, which checkpoints are involved, and which commands should be used, while keeping future agents away from unrelated legacy triplet code.

## Scope

When the user asks about the **soft triplet refinement** model, focus on these files first:

- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_inference.py`
- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_eval.py`
- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_merge_eval_json.py`
- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_fit_temperatures.py`
- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_target_relabel.py`

This guide covers only the soft-refinement path. It does **not** document the old hard cascade training or the broad legacy head zoo in this repo.

## Main Idea

The soft-refinement pipeline is **not** the old hard cascade.

It uses six already-trained heads:

1. Stage-0 clip heads
   - `tool0`
   - `verb0`
   - `target0`
2. Pair-conditioned refinement heads
   - `tool | verb,target`
   - `verb | tool,target`
   - `target | tool,verb`

The important property is:

- routing is done with **soft probabilities**
- refinement does **not** depend on hard argmax labels

In practice:

1. run the encoder once on the clip or evaluation sample
2. compute stage-0 probabilities
3. use those probabilities as routing weights over the conditioned experts
4. blend previous-stage and refined probabilities
5. optionally run a second refinement round

## Hard Rules

- Do not treat the old sequential hard pipeline as the main entrypoint for this work.
- Do not change `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_conditioned_inference.py` unless the user explicitly wants the old hard-conditioned path changed too.
- Do not wander across old `configs/heads/*` trees unless the user asks for older experiments.
- For the current soft work, prefer the `surg_finetuned_phase3_conditioning_study` configs referenced by the soft scripts.
- Preserve backward compatibility:
  - single-machine soft eval must still work
  - sharded two-Spark soft eval must still work
  - old hard pipeline commands must still work
  - identity calibration (no `--calibration-json`) must remain a
    byte-for-byte no-op relative to the pre-calibration pipeline
- Treat the temperature JSON as data, not code: it is regenerated by
  `triplet_soft_refinement_fit_temperatures.py` from the val split.
  Do not hand-edit fitted temperatures unless the user explicitly
  asks for that.
- Do not edit the Tool5 validation CSVs on disk for relabel experiments.
  In-memory target relabeling is opt-in via `--target-relabel-mode` on
  the eval and fit scripts only; the on-disk grouped val CSV and
  metadata JSON stay the single source of truth for training and for
  other head configs.

## Files That Matter

### Inference

- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_inference.py`

This is the main soft-refinement implementation. It defines:

- default configs
- checkpoint fallback logic
- grouped-verb to study-verb adapter logic
- soft routing and probability refinement
- single-video inference path
- the `HeadTemperatures` calibration object and the
  `--calibration-json` CLI flag (logit-space temperature scaling
  applied at all six sigmoid points, identity by default)

### Batch Evaluation

- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_eval.py`

This evaluates the soft model on the clip-level Tool5 validation split and reports:

- exact accuracy
- Macro F1
- Micro Precision
- Micro Recall
- Micro F1
- joint exact tool/verb/target accuracy
- unsupported predicted triplets

It supports:

- single-machine evaluation
- sharded evaluation across multiple machines with
  - `--num-shards`
  - `--shard-index`
- per-head logit-space calibration via `--calibration-json`
- bootstrap confidence intervals on macro/micro F1 via
  `--bootstrap-iterations` (default 1000), `--bootstrap-seed`,
  `--bootstrap-confidence` (default 0.95)
- per-sample predicted/target packed-bit masks under
  `summary["per_sample_masks"]`, used to compute correct cross-shard
  CIs at merge time (suppress with `--no-store-per-sample-masks`)
- a `summary["provenance"]` block with timestamp, hostname, python
  executable, torch version, full `argv`, args dict + sha256, sha256
  for each of the six head checkpoints, the six configs, runtime CSV,
  label CSV, triplet metadata, optional backbone checkpoint, optional
  calibration JSON, plus git HEAD SHA and dirty status
- optional in-memory **target relabel** via `--target-relabel-mode`
  (see `triplet_soft_refinement_target_relabel.py`). When set, the
  evaluator rewrites val target/triplet ground truth and the predicted
  target/triplet masks at metric time only; dropped target classes are
  excluded from Macro and Micro F1 (class-masking, not "penalize as
  false"). The eval JSON includes `summary["settings"]["target_relabel_plan"]`.

### Target relabel (eval / calibration only)

- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_target_relabel.py`

This module is used only by the eval and temperature-fit scripts. It
does not change inference, routing, or head weights, and it does not
write to any file.

Supported mode `tool5_chole_v1` (opt-in):

- drop from evaluation: `suture`, `gut`, `specimen bag` (by class name
  lookup in the target0 config’s `class_names_per_task[0]`)
- merge: `gallbladder wall` → `gallbladder`
- `hepatic pedicle` is not in the Tool5 16-class target list; it is a
  no-op if named in informal requirements

Triplet bits are remapped using the supported set in
`triplet_multilabel_native_tool5_metadata.json`. If a merge would
require a supported triplet that does not exist, that triplet id is
dropped from the relabeled eval (counted in the plan’s diagnostics).

### Merge Utility

- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_merge_eval_json.py`

This merges sharded JSON outputs from the evaluator into one exact final report.

It also:

- deduplicates `per_sample_masks` across shards by `global_indices`
  and re-runs the bootstrap on the union, so cross-shard CIs are
  correct (`--bootstrap-iterations`, `--bootstrap-seed`,
  `--bootstrap-confidence`)
- preserves `provenance` from each shard under `provenance_per_shard`
- can re-emit the merged per-sample masks with
  `--keep-per-sample-masks`

### Temperature Calibration

- `/path/to/phase_triplet_heads_bundle/vjepa2_1/triplet_soft_refinement_fit_temperatures.py`

This fits one positive scalar temperature per head on the Tool5
clip-level validation split by minimizing BCE-with-logits.

Behavior:

- stage-0 heads (`tool0`, `verb0`, `target0`) are fit against the
  clip-level multi-label ground truth in their native space (verb0 in
  the 7-way grouped space)
- conditioned heads are fit only on `(clip, condition_pair)` rows
  whose pair appears in a ground-truth triplet for that clip; the
  supervision target is the remaining axis of those triplets in the
  head's own space (verb in the 6-way study space)
- search is golden-section over `[0.1, 10.0]` by default
- output JSON is consumable directly by `--calibration-json` on both
  the inference and evaluation scripts
- optional `--target-relabel-mode tool5_chole_v1` so temperature fit
  uses the same in-memory relabeled val labels as the relabeled eval
  (keeps calibration and eval consistent; omit for legacy behavior)

The fitted JSON exposes a `temperatures` object keyed by:

- `tool0`, `verb0`, `target0`
- `tool_refine`, `verb_refine`, `target_refine`

Identity (1.0) is the default and matches uncalibrated runs exactly.

## Current Soft Defaults

Recommended current defaults:

- `--refinement-steps 2`
- `--blend-alpha 0.75`
- `--blend-alpha-step2 0.9`
- `--route-topk-tool 2`
- `--route-topk-verb 3`
- `--route-topk-target 4`
- `--routing-probability-source current`
- `--support-constrained-routing-mode hard`
- `--support-aware-triplet-refinement`
- `--decode-topk-tool 3`
- `--decode-topk-verb 3`
- `--decode-topk-target 1`
- `--decode-max-triplets-total 3`
- `--decode-score-min 0.2`

Optional decode-time constraint:

- `--support-aware-triplet-refinement`

This keeps the learned soft heads unchanged, but decodes the final triplet outputs
through the supported Tool5 triplet metadata with a top-k per-tool search over
supported `(tool, verb, target)` candidates. The search scores each supported
triplet with both the final marginal probabilities and the final conditioned
tool/verb/target probabilities, then emits only supported triplets.

Support-constrained routing args:

- `--routing-probability-source {current,stage0}`
  - `current`: re-route on the latest refined marginals at each step
  - `stage0`: keep routing anchored to the original unconditioned marginals
  - recent Tool5 tests favored `current` once step-2 blending was damped
- `--support-constrained-routing`
  - legacy compatibility alias for `--support-constrained-routing-mode hard`
- `--support-constrained-routing-mode {none,soft,hard}`
  - `none`: leave pair-routing weights unchanged
  - `soft`: keep all routed pairs, but downweight unsupported pairs before renormalization
  - `hard`: remove unsupported pairs before renormalization
- `--support-constrained-routing-unsupported-scale 0.1`
  - only used in `soft` mode
  - multiplies unsupported pair weights before renormalization
  - valid range is `[0, 1]`

These routing constraints are applied inside the soft refinement loop before
running the pair-conditioned heads. They are separate from
`--support-aware-triplet-refinement`, which acts only at final triplet decode time.

Multi-step damping arg:

- `--blend-alpha-step2`
  - optional override for the second refinement round only
  - when omitted, step 2 reuses `--blend-alpha`
  - the current best Tool5 result uses `--blend-alpha 0.75 --blend-alpha-step2 0.9`

Optional calibration:

- `--calibration-json /path/to/temperatures.json`

This applies per-head temperature scaling in logit space before the
sigmoid in `compute_soft_refined_probabilities`. Without this flag the
pipeline behaves exactly as before.

Optional bootstrap:

- `--bootstrap-iterations 1000` (default)
- `--bootstrap-seed 20260101` (default)
- `--bootstrap-confidence 0.95` (default)
- `--no-store-per-sample-masks` (only set if disk size matters; needed
  for correct cross-shard CIs)
- optional target relabel: `--target-relabel-mode tool5_chole_v1`
  (in-memory only; no CSV write). Tool and verb metrics are unchanged;
  target and triplet metrics use class-masking for dropped classes. Use
  the same mode when fitting temperatures and when evaluating for
  comparable numbers.

Important behavior:

- if sharding is not passed, evaluator runs on a single machine exactly as before
- if sharding is passed, each machine evaluates a disjoint subset of sample indices
- per-shard bootstrap CIs are produced inside each shard JSON; the
  authoritative combined CI is produced by the merge script over the
  union of per-sample masks

## Latest Routing Conclusions

Current architectural conclusions from the recent support-routing ablations:

1. Support-constrained routing helps, but it should be paired with support-aware decode.
   - merged `routing + decode` beat `decode only` on triplet metrics
   - triplet MicroF1: `42.1775` vs `41.4352`
   - triplet MacroF1: `3.6839` vs `3.5005`
2. Routing-only is not a good final operating point.
   - merged `routing only` fell to triplet MicroF1 `37.3156`
   - it also emitted `143` unsupported predicted triplets
3. With decode held fixed, `hard` routing is currently the best setting.
   - `hard_decode`: triplet MicroF1 `42.1775`, MacroF1 `3.6839`, unsupported triplets `0`
   - `soft_decode` at `--support-constrained-routing-unsupported-scale 0.1`: triplet MicroF1 `42.1091`, MacroF1 `3.6635`, unsupported triplets `0`
4. `soft` routing is still useful to keep in mind.
   - it slightly improved target metrics over `hard`
   - target MicroF1: `54.1894` soft vs `53.9382` hard
   - target MacroF1: `20.0344` soft vs `19.8414` hard
5. Practical recommendation for now:
   - keep `--support-constrained-routing-mode hard` as the Tool5 soft-refinement baseline
   - keep `--support-aware-triplet-refinement` enabled with the current decode settings
   - treat `soft` mode as exploratory unless a future scale sweep such as `0.02` or `0.05` beats `hard`

## Latest Multi-Step Conclusions

Current conclusions from the recent 2-step stabilization tests:

1. The main problem with 2-step refinement was not the support-routing prior itself.
   - using `--routing-probability-source stage0` improved marginal metrics but hurt final triplet MicroF1
   - stage0-routed hard decode reached triplet MicroF1 `40.6073`, below the earlier hard-decode baseline `42.1775`
2. The main issue was that the second pass was overwriting step-1 evidence too aggressively.
   - adding `--blend-alpha-step2 0.9` improved all three marginal heads and improved triplet quality too
3. Current best merged Tool5 result:
   - `--refinement-steps 2`
   - `--blend-alpha 0.75`
   - `--blend-alpha-step2 0.9`
   - `--routing-probability-source current`
   - `--support-constrained-routing-mode hard`
   - `--support-aware-triplet-refinement`
   - `--decode-topk-tool 3 --decode-topk-verb 3 --decode-topk-target 1`
   - `--decode-max-triplets-total 3 --decode-score-min 0.2`
4. Metrics for that current best setting:
   - tool MicroF1 `79.9208`
   - verb MicroF1 `66.4885`
   - target MicroF1 `56.0776`
   - triplet MicroF1 `42.5616`
   - triplet MacroF1 `4.3079`
   - triplet Exact `3.0981%`
   - unsupported predicted triplets `0`
5. Practical next sweep:
   - try `--blend-alpha-step2 0.95` next
   - then try `--blend-alpha-step2 0.85` if needed

## What Future Agents Should Avoid

- Avoid treating the full repo as if all head configs are equally current.
- Avoid changing old hard-cascade scripts unless the user explicitly requests that too.
- Avoid removing the grouped-verb to study-verb adapter just because the label counts differ.
- Avoid running the exact same full evaluator command on both Sparks without sharding. That only duplicates work.
- Avoid reporting merged multi-Spark metrics by hand. Use the merge script.
- Avoid computing combined bootstrap CIs by averaging per-shard CIs.
  Always merge per-sample masks first, then bootstrap on the union via
  the merge script.
- Avoid removing the `provenance` block from the eval JSON. It is the
  only reliable way to tell whether two reported numbers came from the
  same checkpoints, configs, and validation files.
- Avoid hand-editing the Tool5 val CSVs or `triplet_multilabel_val_native_tool5.csv`
  to “apply” the target relabel. Use `--target-relabel-mode` on eval and
  fit so training data and other head trees stay aligned with the
  on-disk files.
